# Сценарии демонстрации кеширования RTK Query (для защиты)

Подключено: `keepUnusedDataFor: 120` (секунд), теги `Profile`, `Car`, `Bookings`, `ParkingPlaces`, `SelectionContext`, инвалидация при мутациях.

## 1. Профиль и бронирования без лишних запросов

1. Войти в систему, открыть **Профиль** — выполняется `getProfile` и `getMyBookings` (первый раз с сети).
2. Перейти на **Главную**, снова открыть **Профиль** в пределах 120 с после ухода со страницы — данные берутся из **кеша**, запрос может не уходить на сервер (в DevTools → Network видно отсутствие повторных GET или статус «из кеша» в Redux DevTools → RTK Query).

## 2. Этаж парковки и список мест

1. Открыть **Выбор места**, выбрать этаж **1** — загрузка `getPlaces(1)`.
2. Переключить этаж на **2**, затем обратно на **1** в течение двух минут — ответ по этажу **1** снова из **кеша** (ключ запроса — номер этажа, тег `ParkingPlaces` с `id` этажа).

## 3. Инвалидация после бронирования

1. На странице выбора места забронировать место — мутация `createBooking` помечает теги `Bookings`, `ParkingPlaces` (этажи), `Profile`.
2. На **Профиле** список сессий **обновится** (новый запрос или перерисовка из инвалидированного кеша) — показать, что данные после действия пользователя **согласованы** с сервером.

## 4. Инвалидация после смены автомобиля

1. В профиле сохранить данные авто — `createCar` / `updateCar` инвалидирует `Car`, `Profile`, `SelectionContext`.
2. Вернуться на **Выбор места** — контекст авто (`getSelectionContext`) и профиль подтянутся заново, тип ТС (например, электро) обновится.

## 5. Redux DevTools (по желанию)

Открыть Redux DevTools → вкладка состояния → `parkingApi` → `queries`: видны `fulfilledTimeStamp`, `data`, время жизни кеша; при мутации — `invalidations` и повторные `pending` для затронутых запросов.
